# Tessellation Shaders

- Only vertex shaders have been available for us to manipulate geometric primitives
- Vertex shaders cannot create additional geometry during their execution
- Only update the data associated with the current vertex they are processing
- OpenGL pipelines contains other shader stages that address limitations
	- Tessellation shaders
		- Generate a mesh of triangles using a patch
- With tessellation specify a patch (ordered list of vertices)
- When a patch is rendered, the tessellation control shaders execute first, operating on patch, and specifying how much geometry should be generated from your patch
- After tessellation control, tessellation evaluation shaders position the vertices of the generated mesh using tessellation coordinates and sends them to the rasterizer
- OpenGL passes the output of the tessellation control shader to the primitive generator, which generates the mesh of the geometric primitives and tessellation coordinates
- Tessellation evaluation shader positions each of the vertices in the final mesh
- Displacement mapping combines texture-mapping for vertices with tessellation shaders

# Tessellation Patches

- Tessellation process uses a new primitive called a patch
- Patches are processed by all the active shading stages in the pipeline
- If any tessellation shaders are active, passing any other type will generate an error
- Patches are an ordered list of vertices
- OpenGL needs to be told how many vertices from your vertex array to use to make one patch
- Patches processed by the same dray call will all be the same size

`void glPatchParameteri(GLenum pname, GLint value)`
- Specifies the number of vertices in a patch using value

_Specifying Tessellation Patches_
```c
GLfloat vertices [][2] = {
	{-0.75, -0.25}, {-0.25, -0.25}, {-0.25, 0.25}, {-0.75, 0.25},
	{ 0.25, -0.25}, { 0.75, -0.25}, { 0.75, 0.25}, { 0.25, 0.25}
};

glBindVertexArray(VAO);
glBindBuffer(GL_ARRAY_BUFFER, VBO);
glBufferData(GL_ARRAY_BUFFER, sizeof(vertices), vertices, GL_STATIC_DRAW);
glVertexAttribPointer(vPos, 2, GL_FLOAT, GL_FALSE, 0, BUFFER_OFFSET(0));
glPatchParameteri(GL_PATCH_VERTICES, 4);
glDrawArrays(GL_PATCHES, 0, 8);
```

- Vertices of each patch are processed by the currently bound vertex shader

# Tessellation Control Shaders

# Controlling Tessellation

- The other function is to specify how much to tesselate the output patch

3 Tessellation domains
1) Quadrilateral
2) Triangle
3) Collection of isolines

Tessellation values
1) Inner tessellation levels
	1) Interior domain is sub-divided
2) Outer tessellation levels
	1) Perimeter of the domain is sub-divided

- Level factors arrays are fixed, the number of values used from those arrays depend on the tessellation domain
- Each level factors specifies how many segments to subdivide a region


**Quad Tessellation**

- The quad domain sub-divides the unit square using all the inner and outer-tessellation levels

![[Pasted image 20250924173051.png]]

_ Tessellation Levels for Quad Domain Tessellation_
```c
gl_TessLevelOuter[0] = 2.0;
gl_TessLevelOuter[1] = 3.0;
gl_TessLevelOuter[2] = 2.0;
gl_TessLevelOuter[3] = 5.0;

gl_TessLevelInner[0] = 3.0;
gl_TessLevelInner[1] = 4.0;
```

- Outer-tessellation corresponds to the number of edges of the perimeter
- Inner-tessellation specify the regions in the horizontal and vertical directions
- Solid circle represents the tessellation coordinates (u, v)


**Isoline Tessellation**

- Isoline domain also generates (u, v) pairs
- Only use two of the outer-tessellation levels

_Tessellation Levels for an Isoline Domain Tessellation_
```c
gl_TessLevelOuter[0] = 6;
gl_TessLevelOuter[1] = 8;
```

- Isolines do not include a tessellated isoline along that tope edge

 - Coordinates related to the three vertices use (a, b, c)

```c
out vec4 color;

{

	float omu = 1 - u;
	float v = gl_TessCoord.y;
	float omv = 1 - v;

		omu * omu * gl_in[0].gl_Position +

		u * v * gl_in[2].gl_Position +

```

# Evaluating Tessellation Coordinates for the Teapot

![[Pasted image 20250925111505.png]]

- Also specify our `quads` domain, spacing options, and polygon face orientation

_The `Main` Routine of the Teapot Tessellation Evaluation Shader_
```c
#version 420 core

layout (quads, equal_spacing, ccw) out;

uniform mat4 MV; // Model-view matrix
uniform mat4 P; // Projection matrix

void main()
{
	vec4 p = vec4(0.0);

	float u = gl_TessCoord.x;
	float v = gl_TessCoord.y;

	for (int j = 0; j < 4; j++){
		for (int i = 0; i < 4; ++i){
			p += B(i, u) * B(j, v) * gl_in[4*j+i].gl_Position;
		}
	}
	gl_Position = P * MV * p;
}
```
- B function is one of the Bernstein polynomials, which is an entire family of mathematical function

![[Pasted image 20250925111911.png]]

_Definition of B(i, u) for the Teapot Tessellation Evaluation Shader_
```c
float
B(int i, float u)
{
	// Binomial coefficient lookup table
	const vec4 bc = vec3(1, 3, 3, 1);
	return bc[i] * pow(u, i) * pow(1.0 - u, 3 - i);
}
```

# Additional Tessellation Techniques

##  View-Dependent Tessellation

- Basing the amount of tessellation on view-dependent parameters
- Level-of-detail scheme based on the distance on the patch from the eye's location

_Computing Tessellation Levels Based on View-Dependent Parameters_
```c
uniform vec3 EyePosition;

void main()
{
	vec4 center = vec4(0.0);
	for (int i = 0; i < gl_in.length(); ++i){
		center += gl_in[i].gl_Position;
	}
	center /= gl_in.length();
	float d = distance(center, vec4(EyePosition, 1.0));
	const float lodScale = 2.5;
	float tessLOD = mix(0.0, gl_MaxTessGenLevel, d * lodScale);

	for (int i = 0; i < 4; ++i){
		gl_TessLevelOuter[i] = tessLOD;
	}

	tessLOD = clmap(0.5 * tessLOD, 0.0, gl_MaxTessGenLevel);
	gl_TessLevelInner[0] = tessLOD;
	gl_TessLevelInner[1] = tessLOD;
	gl_out[gl_InvocationID].gl_Position = gl_in[gl_InvocationID].gl_Position;
}
```

- Cracking is an important issue with tessellation
- A patch is a logical ordering

```c
struct EdgeCenters {
	vec4 edgeCenter[4];
};
```

_Specifying Tessellation Level Factors Using Perimeter Edge Centers_
```c
struct EdgeCenters { vec4 edgeCenter[4]; };

uniform vec3 EyePosition;
uniform EdgeCenters patch[];

void main()
{
	for (int i = 0; i < 4; ++i){
		float d = distance(patch[gl_PrimitiveID].edgeCenter[i],
					vec4(EyePosition, 1.0));
		const float lodScale = 2.5;
		float tessLOD = mix(0.0, gl_MaxTessGenLEvel, d * lodScale);
		gl_TessLevelOuter[i] = tessLOD;
	}

	tessLOD = clamp(0.5 * tessLOD, 0.0, gl_MaxTessGenLevel);
	gl_TessLevelInner[0] = tessLOD;
	gl_TessLevelInner[1] = tessLOD;

	gl_out[gl_InvocationID].gl_Position = gl_in[gl_InvocationID].gl_Position;
}
```

##  Shared Tessellated Edges and Cracking

- In order to prevent cracking along edges between similarly tessellated adjacent patches, the order of accumulation of mathematical operations in the tessellation evaluation shader must also match
- Recognize the outer-patch vertices that contribute to a vertex incident to a perimeter edge, and sort in a predictable manner
- Another technique is applying the `precise` qualifier

##  Displacement Mapping

_Displacement Mapping in `main` Routine of the Teapot Tessellation Evaluation Shader_
```c
#version 420 core
layout (quads, equal_spacing, ccw) out;

uniform mat4 MV;
uniform mat4 P;

uniform sampler2D DisplacementMap;

void main()
{
	vec4 p = vec4(0.0);
	float u = gl_TessCoord.x;
	float v = gl_TessCoord.y;

	for (int j = 0; j < 4; ++j){
		for (int i = 0; i < 4; ++i){
			p += B(i, u) * B(j, v) * gl_in[4*j*i].gl_Position;
		}
	}
	p += texture(DisplacementMap, gl_TessCoord.xy);
	gl_Position = P * MV * p;
}
```
