# Sync Objects

- OpenGL operates in a client-server model
- Server operates asynchronously to the client
- In model graphics, the client is the CPU and the application runs on it, sending commands to the server, which is a high-performance GPU
- Bandwidth is low compared to the throughput and performance
- For max performance, the GPU runs asynchronously to the CPU and can often be several OpenGL commands behind the application
- To achieve a synchronized CPU and GPU use a sync objet, also known as a fence
- A fence is a marker in the stream of commands that can be send along with drawing and state change to the GPU
- Starts life in an unsignaled state and becomes signaled when the GPU has executed it
- Application can look at the state of the fence, and wait for the GPU to have execute before moving on

```c
GLsync glFenceSync(GLenum condition, GLbitfield flags)
```
- Creates a new fence sync object by inserting a fence into the OpenGL stream and returning a handle to the newly created fence

```c
void glGetSynciv(GLsync sync, GLenum pname, GLsizei bufSize, GLsizei *length, GLint *values)
```
- Retrieves the properties of a sync object

```c
GLenum glClientWaitSync(GLsync sync, GLbitfields flags, GLuint64 timeout)
```
- Causes the client to wait for the sync object to become signaled
- Sync objects can only go from the unsigned state into the signaled state

```c
void glDeleteSync(GLsync sync)
```
- Deletes the sync object

- Insert a fence right after the last command that might read from a buffer, and then issues a call to `glClientWaitSync()` before you write into the buffer

_Example Use of a Sync Object_
```c
// Sync object

// Bind a vertex array and draw geometry
glBindVertexArray(vao);
glDrawArrays(GL_TRIANGLES, 0, 30000);

do_something_time_consuming();

// Wait for sync object to be signaled

// Delete the sync object
glDeleteSync(s);

memcpy(data, source_data, source_data_size);

glUnmapBuffer(GL_UNIFORM, source_data_size);)
```

```c

```
- Returns GL_TRUE if sync is the name of an existing sync that has not been deleted

**Advanced**

```c

```
- Causes the server to wait for the sync object indicated by sync to become signaled

## Order-Independent Transparency

- Order-independent transparency is a technique where blending operations are carried out in a manner such that rasterization order is not important
- Geometry must be stored into a fixed, known order before being rendered
- The order in which the geometry is submitted to the GPU is not important
- Arbitrary blending functions can be implemented as all operations are performed in the shader and so programmable blending is achievable


**Principles of Operation**

- The fragment shader builds a set of linked lists into a large, one-dimensional buffer image
- An atomic counter is used to keep track of the total number of fragments appended to the list
- Each pixel on the screen generates a separate linked list of all of the transparent fragments that contribute to it
- The resulting linked lists are interleaved and each pixel has its own head pointer, stored in a 2D image that is the size of the framebuffer

<img src="/images/Pasted image 20250930092147.png" alt="image" width="500">

- Once the linked list data structure have been built, a second pass is performed over the entire framebuffer
- Fragment shader traverses the linked list corresponding to its pixel and sorts all the fragments in the list in order of depth


**Initialization**

**Sorting and Blending**

-  Render a full-screen quad using a fragment shader that will read from the list corresponding to its output pixel
- Sort all of the fragments into order
- Perform the blending operation

_Main Body of Final Order-Independent Sorting Fragment Shader_
```c
#version 420 core
// Head pointer 2D buffer
uniform sampler2F head_pointer_image;

// Linked list 1D buffer
uniform samplerBuffer list_buffer;

#define MAX_FRAGMENTS 15

// Small buffer to hold fragments
uvec4 fragments[MAX_FRAGMENTS];

layout (location = 0) out vec4 output_color;

void main(void)
{
	int frag_count;
	// Traverse the list and build an array of fragments
	frag_count = build_local_fragment_list();
	// Sort the array in depth order
	sort_fragment_list(frag_count);
	// Blend the sorted fragments together
	output_color = calculate_final_color(frag_count);
}
```

- The fragment lists only needs to be large enough to store the average number of fragments per-pixel, but the maximum overdraw supported can be much larger

_Traversing Linked-Lists In a Fragment Shader_
```c
// Traverse the linked list
int build_local_fragment_list(void)
{
	uint current;
	int frag_count = 0;

	// Get initial head pointer from the head-pointer image
	current = texelFetch(head_pointer_image,
				ivec2(gl_FragCoord.xy), 0);
	// Stoarge
	while (current != 0xFFFFFFFF & frag_count < MAX_FRAGMENTS)
	{
		// Read an item from list
		item = texelFetch(list_buffer, current);
		current = item.x;
		uvec4 fragments[frag_count] = item;
		frag_count++;
	}
	// Return the fragment count
	return frag_count;
}
```

- After the local array of fragments has been built it is sorted in order of depth using `sort_fragment_list`
- Implements a simple bubble sort algorithm

_Sorting Fragments into Depth Order for OIT_
```c
// Bubble sort
void sort_fragment_list(int frag_count)
{
	int i, j;
	for (i = 0; i < frag_count; j++)
	{
		float depth_i = uintBitsToFloat(fragments[i].z);
		float depth_j = uintBitsToFloat(fragments[j].z);
		// Compare depth
		if (depth_i > depth_j)
		{
			// Swap fragment in the array
			uvec4 temp = fragments[i];
			fragments[i] = fragments[j];
			fragments[j] = temp;
		}
	}
}
```
