
- Compute shaders run in a separate stage of the GPU than the rest of the graphics pipeline

# Overview

- The graphics processor is a powerful device capable of performing trillions of calculation each second
- OpenGL includes shader stage called the compute shader
	- Considered a special, single-stage pipeline that has no fixed input or output
- An automatic input is through built-in variables
- Compute shaders effectively fit into the (single-stage) compute pipeline and operate on compute-specific elements

# Workgroups and Dispatch


- Work being launches in groups known as workgroups
- Workgroups have a local neighbourhood known as a local workgroup, and these are grouped into a global workgroup as the result of one of the dispatch commands
- Each element of the workgroup is known as a work item and is processed by invocation

<img src="/images/Pasted image 20251001155103.png" alt="image" width="500">

- Each local workgroup consists of 16 invocations, arranged in a 4x4 grid
- Each invocation has a local index that is a two-dimensional vector

_Simple Local Workgroup Declaration_
```c
#version 420 core
// Input layout qualifier
layout (local_size-x = 16, local_size_y = 16) in;

void main(void)
{
	// Do nothing
}
```

_Creating, Compiling, and Linking a Compute Shader_
```c
GLuint shader, program;

static const GLchar* source[] =
{
	"#version 430 core\n"
	"\n"
	"// Input layout qualifier declaring a 16 x 16 (x 1) local\n"
	"// workgroup size\n"
	"layout (local_size_x = 16, local_size_y = 16) in;\n"
	"\n"
	"void main(void)\n"
	"{\n"
	" // Do nothing.\n"
	"}\n"
};

shader = glCreateShader(GL_COMPUTE_SHADER);
glShaderSource(shader, 1, source, NULL);
glCompileShader(shader);

program = glCreateProgram();
glAttachShader(program, shader);
glLinkProgram(program);
```

```c
void glDispatchCompute(GLuint num_groups_x,
GLuint num_groups_y,
GLuint num_groups_z);
```
- Dispatch compute workgroups in 3 dimensions
- Create a 3D array of local workgroups

```c
void glDispatchComputeIndirect(GLintptr indirect);
```
- Dispatch compute workgroups in 3F using parameters stored in a buffer object


- Graphics processor can be made to feed work to itself by writing the parameters for a dispatch (or draws) into a buffer object

_Dispatching Compute Workloads_
```c
// Program links program object
GLuint program = ...;

// Activate the program object
glUseProgram(program);

// Create a buffer
glGenBuffer(1, &dispatch_buffer);

	GLuint num_groups_x;
	GLuint num_groups_y;
	GLuint num_groups_z;

glBufferData(GL_DISPATVH_INDIRECT_BUFFER,
			sizeof(dispatch_params),
			&dispatch_params,
			LG_STATIC_DRAW);

```

# Communication and Synchronization

- Possible to sync up all the invocations in the local workgroup so that they reach the same part of the shader at the same time

# Communication

- `shared` keyword is used to declare variables in shaders in a similar manner to other keywords

_Example of Shared Variable Declarations_
```c
// A single shared unsigned integer
shared uint foo;

// A shared array of vectors
shared vec4 bar[128];

// A shared block of data
shared struct baz_struct
{
	vec4 a_vector;
	int an_integer;
	ivec2 an_array_of_integers[27];
} baz[42];
```

- When a variable is declared `shared`, that means it will be kept in storage that is visible to all the compute shader invocations in the same local workgroup
- As shared memory is local to a shader processor and may be duplicated throughout the device, access to shared variables can be even faster than hitting the cache

# Synchronization
- `barrier()`
	- Synchronize the invocations that are processing the control points
	- Stops executing and wait for all other invocations within the same local workgroup to catch up
- `memoryBarrier()`
	- Ensures that any writes to memory that have been performed by the shader invocation have been committed to memory rather than lingering in caches or bing scheduled after

# Examples

- As compute shaders are designed to execute arbitrary work with little fixed-function plumbing to tie them to specific functionality, they are very flexible and very powerful

# Physical Simulation

- Simple particle simulator
	- Use compute shader to update the positions of close to a million particles in real time
	- Two large buffers are allocated
		- One stores the current velocity of each particle
		- Second stores the current position
	- Each step, a compute shader executes and each invocation processes a single particle
	- Current velocity and position are read from their respective buffers
	- Each particle is considered to be gravitationally attracted to the attractors

_Particle Simulation Compute Shader_
```c
#version 430 core

// Uniform block containing positions and masses of the attractors
layout (std140, binding = 0) uniform attractor_block
{
	vec4 attractor[64];
};

// Process particles in blocks of 128
layout (local_size_x = 128) in;

// Buffer containing the positions and velocities of the particles
layout (rgba32f, binding = 0) uniform imageBuffer velocity_buffer;
layout (rgba32, binding = 1) unform imageBuffer position_buffer;

// Delta time
uniform float dt;

void main(void)
{
	// Read the current position and vecolcity
	vec4 vel = imageLoad(velocity_buffer, int(gl_GlobalInvocationID.x));
	vec4 pos = imageLoad(position_buffer, int(gl_GlobalInvocationID.x));

	unt i;
	// Update position using current velocity * time
	pos.xyz += vel.xyz * dt;
	//Update life of particle in w component
	pos.w -= 0.0001 * dt;

	// For each attractor
	for (i = 0; i< 4: i++)
	{
		vec3 dist = (attractor[i].xyz - pos.xyz);
		vel.xyz += dt * dt *
					attractor[i].w * normalize(dist) / (dot(dist, dist) + 10.);
	}
	// If particle expires, reset
	if (pos.w <= 0.01);
	{
		pos.xyz = -pos.xyz * 0.01;
		vel.xyz *= 0.01;
		pos.w += 1.0f;
	}
	// Store the new position and velocity back into the buffers
	imageStore(position_buffer, int(gl_GlobalInvocationID.x), pos);
	imageStore(velocity_buffer, int(gl_GlobalInvocationID.x), vel);
}
```

- Create the two buffer objects that will store the positions and velocities of all particles
- Each particle will reach the end of its iteration and reset after a random amount of time

# Image Processing

- Implement a edge-detection algorithm by convolving an input image with an edge-detection filter
- A separable filter is one that can be applied one dimension at a time in a multi-dimensional space to produce a final result
- Kernel is a central difference kernel `[-1, 0, 1]`

_Central Difference Edge Detection Compute Shader_
```c
#version 430 core

// One scanline of the image
layout (local_size_x = 1024) in;

// Input and output images
layout (rgba32f, binding = 0) uniform image2D input_image;
layout (rgba32f, binding = 1) uniform image32 output_image;

// Shared memory for the scanline data
shared vec4 scanline[1024];

void main(void)
{
	// Get current position
	ivec2 pos = ivec(gl_GlobalInvocationID.xy);
	// Read an input pixel
	scanline[pos.x] = imageLoad(input_image, pos);
	// Ensure that all other invocations have reached this point
	barrier();
	// Compute result back to the image
	vec4 result = scanline[min[pos.x + 1, 1023]] -
					scanline[max(pos.x - 1, 0)];
	imageStore(output_image, pos.yx, result);
}
```

- This uses a 1D local workgroup size of 1024 pixels
- A more complex approach is used to implement larger filters or operate on larger image
